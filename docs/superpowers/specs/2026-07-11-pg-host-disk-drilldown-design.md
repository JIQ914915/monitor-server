# PostgreSQL 主机磁盘告警下钻完善设计

## 目标

修正 PostgreSQL「主机磁盘空间不足」事件分析页的趋势取数，并新增 PostgreSQL 专属知识库文章。成功标准：磁盘使用率、磁盘 IO 繁忙度按分钟数据展示，实例总容量按小时数据展示；接口错误与真实无数据能够区分；画像内容和知识库不再出现 MySQL 术语或命令。

## 根因

主机指标由主机采集器按关联实例扇出写入 `metric_data_1m`，因此 `host.disk.usage_max` 和 `host.diskio.util_max` 应按 `1m` 查询。`pg.capacity.total_size_bytes` 由 PostgreSQL 容量采集写入 `metric_data_1h`，但事件分析页的小时级指标集合只包含 MySQL 指标，导致 PG 容量被错误查询分钟表。

页面目前捕获趋势接口的所有异常后直接写入空数组，统一显示“窗口内暂无数据”，使接口失败、参数错误和真实无数据无法区分，增加排障难度。

## 设计

### 画像与取数

在下钻画像的 `related_metrics` 条目中增加可选 `frequency` 字段，允许值为 `1m` 或 `1h`。前端优先使用画像声明；未声明时保持现有默认逻辑，兼容历史画像。

`pg_host_disk` 使用以下指标：

- `host.disk.usage_max`：`1m`，告警主指标；
- `host.diskio.util_max`：`1m`，辅助判断磁盘繁忙；
- `pg.capacity.total_size_bytes`：`1h`，转换为 GB 展示。

事件分析页按每个指标的频率分别确定窗口：分钟级使用告警前后 30 分钟，小时级使用告警前后 12 小时。历史同期对比复用相同频率和窗口，避免两个区块口径不一致。

### 错误状态

为每个趋势指标维护独立错误文本。接口请求失败时图表展示“趋势查询失败”，并保留简短、非敏感的错误信息；只有请求成功且点集为空时才显示“窗口内暂无数据”。页面不打印请求头、Cookie、Token 或数据库连接信息。

### 知识库

新增《PostgreSQL 主机磁盘空间不足排查与处置》，内容包括：

- 确认告警挂载点与 PGDATA、`pg_wal`、日志、归档、备份目录的关系；
- 使用数据库视图与主机只读命令定位容量来源；
- 排查归档失败、失效复制槽、WAL 堆积、临时文件和表膨胀；
- 明确禁止手工删除 `pg_wal`、数据文件和未知复制槽；
- 分级给出应急释放空间、归档治理与磁盘扩容建议。

迁移将文章写入 `knowledge_article`，并将文章 ID 关联到 `pg_host_disk` 画像。迁移保持幂等；已存在同标题文章时不重复创建。

## 兼容性与范围

不修改采集器和时序表结构，不引入新依赖。MySQL 下钻画像继续使用现有默认识别逻辑。仅调整事件分析页的指标频率解析、错误展示和 PostgreSQL 磁盘画像/知识内容。

## 测试与验证

- 前端单元测试验证画像 `frequency=1h` 时调用趋势接口使用 `1h` 和 12 小时窗口；`1m` 使用 30 分钟窗口。
- 前端单元测试验证接口异常显示“查询失败”，成功空数组显示“窗口内暂无数据”。
- 后端/迁移检查验证 `pg_host_disk` 仅匹配 PostgreSQL，且知识文章关联存在。
- 运行前端类型检查、相关前端测试和后端相关 Maven 测试。
- 部署后选取一个包含三类历史数据的 PG 事件，核对三张图与数据库同窗口聚合结果一致。

## 非目标

本次不调整主机采集周期、历史保留策略、告警阈值或其他数据库类型的知识库内容。
